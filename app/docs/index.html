iot-bee · servicio de ingestión IoT

API HTTP en Actix-web que configura pipelines; cada pipeline corre como un árbol de actores Actix que consumen de un broker (RabbitMQ vía lapin), procesan y persisten en un almacén externo. La configuración interna se guarda en PostgreSQL (sqlx).

Visión general

iot-bee separa estrictamente configuración y ejecución. La capa HTTP permite definir las piezas (esquemas de validación, fuentes de datos, almacenes, grupos, pipelines) y la capa de actores pone esas piezas en marcha como procesos vivos: cada pipeline puede tener varias réplicas trabajando en paralelo.

1 · Define

El usuario llama al API REST para registrar fuentes, almacenes y pipelines. Los datos se persisten en PostgreSQL.

2 · Despliega

Un PipelineSupervisor arranca N réplicas; cada réplica es un trío de actores: consumer → processor → store.

3 · Observa

El supervisor expone stop_all, restart_all y status_all; el logging usa tracing con un wrapper propio (AppLogger).

Estado del proyecto

El proyecto está en fase de construcción. La capa de réplica (consumer/processor/store + el supervisor de un pipeline) está integrada y cubierta por tests. La orquestación multi-pipeline aún no está cableada al binario.

Inicio rápido

Requisitos

  • Rust estable (edición 2024).
  • PostgreSQL accesible; un broker AMQP accesible si se quieren correr pipelines reales.
  • Variable DATABASE_URL definida (se carga vía dotenv).

Aplicar migraciones y arrancar

# 1) preparar .env
echo 'DATABASE_URL=postgres://iot_bee:iot_bee_dev@localhost:5432/iot_bee' > .env

# 2) aplicar migraciones (no se ejecutan solas)
sqlx migrate run

# 3) levantar servidor + Swagger
cargo run
# → http://127.0.0.1:8080
# → http://127.0.0.1:8080/swagger-ui/

Tests

cargo test                                  # todos los binarios de test
cargo test --test pipeline_integration      # un binario (ver [[test]] en Cargo.toml)
cargo test --test supervisor_actor some_fn  # filtrar por nombre dentro del binario
RUST_LOG=iot_bee=debug cargo run            # subir verbosidad
Convención de tests
Los tests viven en test/ (singular, no tests/) y se registran explícitamente como [[test]] en Cargo.toml. Si añades un archivo nuevo y no lo registras, Cargo no lo descubre.

El patrón Bridge

Un Bridge es un wrapper cloneable sobre Addr<Actor> que implementa un trait‑port del dominio. Esto evita que los tipos de Actix se filtren fuera del módulo de adaptadores: los actores se conectan entre sí a través de traits, no de tipos concretos.

Protocolo de ciclo de vida

Definido en pipeline_actor_module/general_messages.rs. Un único Message se reusa en los tres actores: 

enum ActorActions { Start, Stop, Restart, Status }
enum ActorStatus  { Running, Stopped, Restarting, Failed }

struct SendActorActionMessage(ActorActions);  // Message
type SendActorActionMessageResult =
    Result<ResponseActorActionMessage, IoTBeeError>;

Apagado limpio
Cuando el DataSource retorna, su Sender<DataConsumerRawType> se descarta. Eso cierra el canal mpsc; el consumer detecta el cierre y se detiene; los siguientes actores se quedan sin tráfico y eventualmente paran. Este es el patrón usado por los tests de integración.

Invariante crítico de concurrencia

⛔ No sostener un RwLockGuard a través de un .await
Hacerlo bloquea el runtime de tokio. ReplicaRegistry contiene un RwLock<Vec<Arc<PipelineAbstractionController>>>; los handlers async deben clonar los Arc en la parte síncrona y soltar el guard antes de cualquier punto de suspensión.

❌ Mal · deadlock latente

async fn stop_all_bad(reg: &ReplicaRegistry) {
    let guard = reg.read_lock()?;       // 🔒
    for r in guard.iter() {
        r.stop().await;                  // ⛔ guard sigue vivo
    }
}

✓ Bien · usando all_arcs()

async fn stop_all_ok(reg: &ReplicaRegistry) {
    let arcs = reg.all_arcs()?;          // 🔓 guard liberado
    for r in arcs {
        r.stop().await;                  // ✓ sin guard sostenido
    }
}

Documentado en los comentarios de pipeline_supervisor.rs y pipeline_abstraction.rs. Si añades un nuevo handler de ciclo de vida, respeta este patrón.

Toda la pila converge en IoTBeeError (src/domain/error.rs), un enum que envuelve los errores por área. adapters/api/error.rs implementa actix_web::ResponseError: los handlers HTTP pueden retornar Result<HttpResponse, IoTBeeError> y obtener status + JSON automáticamente.

Logging

Wrapper sobre tracing: crate::logging::AppLogger. La convención es declarar al inicio de cada módulo: 

static LOGGER: AppLogger = AppLogger::new(
    "iot_bee::ruta::al::modulo",
);

LOGGER.info("evento");

El wrapper anexa target, caller_file y caller_line a cada record. El filtro por defecto es iot_bee=info,actix_web=info; se sobrescribe con RUST_LOG.

Idioma
Comentarios y mensajes de log están mayoritariamente en español. Mantener esa convención al editar código adyacente.

Persistencia

PostgreSQL con sqlx (runtime tokio). Pool con max_connections=10, acquire_timeout=5s, idle_timeout=300s, max_lifetime=1800s (ver infrastructure/persistence/connection.rs).

Glosario

Pipeline
Configuración persistida en PostgreSQL que ata data source + validation schema + data store. Cuando se arranca, se materializa como un árbol de actores.
Réplica
Una instancia activa de un pipeline = un trío consumer/processor/store. Un pipeline puede tener varias.
Bridge
Wrapper cloneable de Addr<Actor> que implementa un trait‑port del dominio. Aísla a Actix del resto del código.
Port (puerto)
Trait en domain/inbound o domain/outbound que la capa exterior debe implementar.
Adapter (adaptador)
Implementación concreta de un puerto en la capa infrastructure o adapters.
